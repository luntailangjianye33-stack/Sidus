# Sidus 提出補足資料

この資料は、Sidus の設計コンセプト、市場分析、稼働手順、設計判断の履歴をまとめた提出補足資料です。
README は起動方法と機能概要、この資料は「なぜこの形にしたか」を説明する目的で分けています。

## 1. プロダクトコンセプト

Sidus は、LLMを活用した ES添削・ブラッシュアップツールです。
単にESを書き換えるのではなく、ユーザーがAIの調査・評価・提案を確認し、自分の判断を介在させながら最終稿を作る Human-in-the-Loop 型のWebプロトタイプとして設計しました。

中心に置いた価値は次の3点です。

- AIの出力を鵜呑みにしない
- 企業理解と文章添削を分離して確認できる
- 提案の採用・却下・編集をユーザーが決められる

ES添削では、AIが自然な文章を生成できるだけでは不十分です。
企業情報の誤り、本人らしさの消失、参考ESの安易な模倣、根拠のない断定が混ざると、ユーザーは最終稿を信頼できません。
そこでSidusでは、AIを「代筆者」ではなく「調査・レビュー・提案を行う補助者」として扱っています。

## 2. 市場・競合分析

既存のES支援サービスは、大きく3つに分けられます。

### ES例文・就活メディア

ワンキャリア、外資就活、就活会議などは、通過ESや体験談を大量に閲覧できる点が強みです。
一方で、ユーザーは自分の経験や志望企業に合わせて情報を取捨選択する必要があります。
例文の模倣に寄ると、本人性や企業固有性が弱くなるリスクがあります。

### 一般的な生成AIチャット

ChatGPTなどの汎用LLMは、文章の自然さや構成改善に強みがあります。
ただし、企業情報の出典、レビュー根拠、提案の採用判断が画面上で分離されていないため、ユーザーがどこを信頼してよいか判断しづらい課題があります。

### ES添削サービス

人力添削やAI添削サービスは、修正案を提示する点では有効です。
一方で、企業調査、参考ESの使い方、提案ごとの根拠確認、最終稿への反映管理までを一連の作業空間として扱うものは限定的です。

Sidus はこの隙間を狙い、ES添削を「一回の生成」ではなく、次のような編集ワークフローとして設計しています。

1. 前提情報を入力する
2. 企業情報を調査し、採用可否を確認する
3. 参考ESから構成・語彙・弱い表現だけを抽出する
4. ES本文を複数観点でレビューする
5. 提案ごとに根拠を確認し、採用・却下・編集する
6. 最終稿をユーザーが確定する

## 3. 課題要件への対応方針

課題では、LLMを活用したES添削・ブラッシュアップツールのWebプロトタイプが求められていました。
Sidusでは要件を次のように解釈しました。

- ES原稿入力
  - テキスト入力、PDF / Markdown / テキストファイル抽出に対応
- 添削条件入力
  - 業界、企業名、職種、目標文字数、本人文脈、参照URLを入力可能
- LLMレビュー
  - JSON形式で評価、根拠、提案、最終稿候補を取得
- ユーザー確認UI
  - 企業理解、参考ES、レビュー、提案、最終稿を段階的に確認可能
- テストデータ
  - 個人情報を含まない5パターンの架空ESを用意

## 4. 設計の特徴

### 4.1 企業調査を証拠台帳として扱う

企業情報取得は、LLMに自由生成させるのではなく、source / claim 単位で管理します。

- source
  - URL、タイトル、ドメイン、取得種別、本文抜粋
- claim
  - 正式名称、所在地、証券コード、事業理解、ES論点など
- verification
  - 確認済み、要確認、未確認、矛盾あり

この設計により、企業情報が取得できなかった場合や根拠が弱い場合に、UI上でユーザーが判断できます。

### 4.2 企業区分の切替

国内企業と外資ブランドでは、信頼すべき情報源が異なります。

- 国内企業
  - 公式会社概要、日経会社情報、IR、法人番号、公的DBを重視
- 外資ブランド
  - 公式グローバルサイト、公式日本サイト、公式採用情報を重視
  - 日本法人DBは補助扱い

Goldman Sachs のような外資系企業では、日本法人DBだけを見ると別法人や関連会社に寄る可能性があります。
そのため、ユーザーが国内企業 / 外資ブランドを切り替えられるUIを用意しました。

### 4.3 参考ESは本文ではなく評価基準として使う

通過ESの本文をそのまま再利用すると、盗用や類似表現のリスクがあります。
Sidusでは、参考ESから次の要素だけを抽出します。

- 通過ESの型
- 強い語彙・言い回し
- 弱い汎用表現
- 構成ヒント

これにより、参考ESを「コピー元」ではなく「レビュー基準」として扱います。

### 4.4 レビュー項目の詳細化

ESレビューは、総評だけではユーザーが改善行動に移りづらいため、以下を項目ごとに表示します。

- 対象文
- 評価理由
- 根拠
- 減点理由
- 修正方向
- 良い点
- 弱い点

評価項目をクリックすると詳細が開き、なぜその評価になったかを確認できます。

### 4.5 提案の採用をユーザー判断にする

提案は自動で最終稿に反映しません。
ユーザーが修正前・修正案・根拠を見て、採用、却下、編集反映を選べます。

## 5. アーキテクチャ概要

主な構成は以下です。

- `src/app/page.tsx`
  - UI全体、画面フロー、保存、提案採用、最終稿編集
- `src/app/api/company-research/route.ts`
  - 企業情報調査、source manifest、claim生成、企業理解
- `src/app/api/benchmark-research/route.ts`
  - 参考ESベンチマークの生成
- `src/app/api/review/route.ts`
  - ESレビュー、評価項目、提案、最終稿候補の生成
- `src/app/api/corporate-number-resolve/route.ts`
  - 国税庁法人番号Web-API連携用ルート
- `src/app/api/company-logo/route.ts`
  - 企業ロゴ候補の取得
- `src/data/sample-essays.ts`
  - 提出用サンプルES
- `src/types/sidus.ts`
  - 主要データ型

## 6. 稼働手順

### 6.1 セットアップ

```bash
npm install
cp .env.example .env.local
```

`.env.local` にOpenAI APIキーを設定します。

```env
OPENAI_API_KEY=your_rotated_key_here
OPENAI_MODEL=gpt-4o-mini
```

法人番号Web-APIを使う場合は、国税庁から発行されたアプリケーションIDも設定します。

```env
NTA_CORPORATE_NUMBER_APP_ID=your_app_id_here
```

### 6.2 開発サーバー起動

```bash
npm run dev
```

ブラウザで開きます。

```text
http://localhost:3000
```

### 6.3 検証

```bash
npm run lint
npm run build
```

## 7. デモ手順

推奨デモ企業は、東京エレクトロン株式会社 / AI・ソフトウェア系エンジニアです。

1. 保存ファイルからTELサンプルを開く
2. 前提情報を確認する
3. 企業調査を実行する
4. 日経会社情報、公式会社概要、証拠台帳を確認する
5. 企業理解を採用する
6. 参考ESベンチマークを確認する
7. レビューを実行する
8. 評価項目の詳細を確認する
9. 提案画面で修正前 / 修正案 / 根拠を確認する
10. 提案を採用または編集する
11. 最終稿で文面を確認し、保存またはコピーする

## 9. 設計履歴

### 初期方針

最初は、ES本文と企業名を入力し、LLMがレビューと改善案を返す単純な構成でした。
しかし、実際に複数企業で試すと、企業情報の誤取得、汎用的なレビュー、AIらしい提案文が目立ちました。

### 企業情報取得の改善

三菱UFJ銀行、東京エレクトロン、鹿島建設、Goldman Sachs などで検証し、以下の課題を確認しました。

- 別法人の混入
- 公式ではないDB情報への過度な依存
- 日経会社情報を拾える企業と拾えない企業の差
- ナビ文やFAQ文の混入
- 外資ブランドで日本法人DBに引っ張られる問題

これを受け、企業調査を「LLMエージェント」ではなく、証拠台帳、固定抽出、claim検証、ユーザー採用の流れに変更しました。

### レビュー・提案の改善

レビューが薄い場合、ユーザーは「普通のLLMでよい」と感じます。
そのため、星評価だけでなく、対象文、評価理由、根拠、減点理由、修正方向を表示する設計にしました。

提案についても、採用するとすぐ最終稿に飛ぶ挙動をやめ、提案画面で比較・確認してから採用できるようにしました。

### UIの改善

作業順序を明確にするため、左サイドバーに以下の流れを固定しました。

```text
前提情報 -> 企業調査 -> 参考ES -> レビュー -> 提案 -> 最終稿
```

未完了の工程は控えめに表示し、次に進むべき工程を目立たせています。
保存ファイルもサイドバーに統合し、作業途中のESやサンプルに戻りやすくしました。

## 10. 今後の改善案

提出時点では、企業情報取得の完全な汎用化はまだ課題です。
今後は次の改善が有効です。

- 20社以上のGolden Datasetを作る
- 企業調査APIの自動評価スクリプトを作る
- 正しい公式ドメイン、証券コード、法人番号、日経会社情報URLの取得率を測る
- 別法人混入、ロゴ誤取得、ナビ文混入を失敗分類として記録する
- 検索ランキングの重みを評価結果にもとづいて調整する

企業調査は、検索回数を増やせば安定するものではなく、証拠管理と評価基盤が重要だと判断しました。

## 12. 既知の制約

- 外部サイトのHTML構造やアクセス可否によって情報取得品質が変わります。
- 国税庁法人番号Web-APIを使うにはアプリケーションIDが必要です。
- 企業名だけでは同名・類似法人が混ざる可能性があります。
- 公式URL、証券コード、日経会社情報URL、法人番号を入力すると企業識別が安定します。
- 保存データはブラウザのローカルストレージに保存されます。
